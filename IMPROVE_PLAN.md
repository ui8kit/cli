## Расширенные предложения по улучшению (ориентир — shadcn)

### 1. Архитектура команд

**`diff` — сравнение локальных компонентов с реестром**

У shadcn есть `diff` команда, которая показывает разницу между установленным компонентом и версией из registry. Это критически важно для обновлений.

```
ui8kit diff              # показать все компоненты с обновлениями
ui8kit diff button       # показать diff для конкретного компонента
```

**`list` / `search` — просмотр доступных компонентов**

Сейчас пользователь не знает, какие компоненты доступны, пока не попробует `add`. Нужна команда:

```
ui8kit list              # все компоненты из registry
ui8kit search dialog     # поиск по имени/описанию
```

**`info` — диагностика окружения**

```
ui8kit info              # версия CLI, Node, пакетный менеджер, конфиг, registry URL
```

**`update` — обновление компонентов**

Отдельная команда (или алиас к `add --overwrite`):

```
ui8kit update button     # обновить компонент
ui8kit update --all      # обновить все
```

---

### 2. Preflight система (как у shadcn)

Сейчас `validateComponentInstallation()` всегда возвращает `{ isValid: true }`. Нужна реальная preflight-проверка перед `add`:

- Существует ли директория проекта
- Есть ли `package.json`
- Есть ли `ui8kit.config.json` (если нет — предложить `init`)
- Совместима ли версия Node
- Есть ли доступ к CDN

---

### 3. Трансформация файлов (Transform Pipeline)

Shadcn использует `ts-morph` для AST-трансформации компонентов при установке. Рекомендую аналогичный pipeline:

- **transformImport** — перезапись путей импортов на основе `aliases` из конфига
- **transformRsc** — добавление `"use client"` для React Server Components (если пользователь использует Next.js)
- **transformCleanup** — удаление лишних комментариев, форматирование

Это позволит компонентам корректно работать при любой структуре проекта, а не только при дефолтной.

---

### 4. Рекурсивное разрешение `registryDependencies`

Сейчас CLI устанавливает только прямые npm-зависимости. Но shadcn рекурсивно разрешает `registryDependencies` — когда компонент `dialog` зависит от `button`, `button` автоматически устанавливается. Нужен:

- Рекурсивный обход дерева зависимостей
- Топологическая сортировка (алгоритм Кана)
- Дедупликация
- Обнаружение циклических зависимостей

---

### 5. Типизированная система ошибок

Сейчас ошибки обрабатываются через `catch(error)` без классификации. У shadcn есть типизированные ошибки:

```typescript
class RegistryNotFoundError extends Error { /* 404 */ }
class RegistryFetchError extends Error { /* network */ }
class ConfigParseError extends Error { /* invalid config */ }
class RegistryParseError extends Error { /* invalid JSON */ }
```

Каждая ошибка содержит `message`, `cause` и `suggestion` — так пользователь получает actionable feedback, а не сырой стектрейс.

---

### 6. `--dry-run` с реальным предпросмотром

Флаг `--dry-run` уже есть в `add`, но нужно сделать его полноценным:

- Показать какие файлы будут созданы/перезаписаны
- Показать какие npm-зависимости будут установлены
- Показать какие registry-зависимости будут подтянуты
- Показать diff для существующих файлов

---

### 7. Конфигурация через cosmiconfig

Сейчас конфиг ищется вручную в 3 местах. Shadcn использует `cosmiconfig` — стандарт индустрии для поиска конфигурации:

```typescript
const explorer = cosmiconfig("ui8kit", {
  searchPlaces: ["ui8kit.config.json", ".ui8kitrc", ".ui8kitrc.json"]
});
```

Это даст автоматический поиск вверх по дереву директорий и поддержку monorepo.

---

### 8. Поддержка monorepo

Сейчас CLI не понимает workspace-структуру. Нужно:

- Флаг `--cwd <path>` для всех команд (есть только в `scan`)
- Обнаружение workspace root (`pnpm-workspace.yaml`, `workspaces` в `package.json`)
- Возможность указать `componentsDir` относительно workspace package

### 9. Улучшение `init`

Текущий `init`:
- Спрашивает только TypeScript и overwrite
- Не предлагает выбор базовых цветов/стилей
- Не проверяет существующий проект

Улучшенный `init` (по аналогии с shadcn):

```
? Which framework do you use? › Vite + React / Next.js / Remix
? TypeScript? › Yes
? Where is your global CSS file? › src/index.css
? Configure import aliases? › @/components
? Which base color? › neutral / gray / zinc / stone / slate
```

И добавить `--defaults` / `-y` для полного скипа промптов с дефолтными значениями.

---

### 10. Интерактивный выбор компонентов

Когда `add` вызывается без аргументов, shadcn показывает multiselect-список всех компонентов. Сейчас CLI выдаёт ошибку. Нужно:

```
? Which components would you like to add?
  ◯ accordion
  ◯ alert
  ◉ button
  ◉ dialog
  ◯ input
```

### 11. Тесты

В проекте нет ни одного теста. Минимальный набор:

- **Unit-тесты**: валидация схем, резолвинг путей, определение пакетного менеджера
- **Integration-тесты**: `init` создаёт правильную структуру, `add` скачивает и записывает компоненты
- **E2E**: полный цикл `init` → `add button` → проверка файлов

Использовать `vitest` (уже Bun-совместим, быстрый).

---

### 12. Кеширование registry

Каждый `add` заново скачивает `index.json`. Нужен локальный кеш:

- Кешировать `index.json` и компоненты в `~/.ui8kit/cache/`
- TTL: 1 час (или `--no-cache`)
- Команда `ui8kit cache clear`

---

### 13. Логирование и verbose-режим

Добавить глобальный флаг `--verbose` / `-v`:

- Показывать URL-ы запросов к CDN
- Показывать полные пути файлов
- Показывать время выполнения каждого шага

---

### 14. Graceful degradation для CDN

Сейчас есть 3 CDN-источника, но fallback работает только в `retry-api.ts` (только с флагом `--retry`). Нужно:

- Автоматический fallback на следующий CDN при ошибке (всегда, не только с `--retry`)
- Проверка доступности CDN при `init`
- Таймаут для запросов (сейчас нет)

---

### 15. Прогрессивная обратная связь

Улучшить UX при установке нескольких компонентов:

Вместо одного спиннера на весь процесс.
